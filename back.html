Oracle → MySQL 자동 복제 배치를
데드락 없이 안정적으로 구동

문제: Oracle의 배치 대상 테이블 목록을 읽어와, 앞서 개발한 MySQL 메타 구조 역공학 프로그램(Python)을 응용해 테이블별 ETL 작업을 자동 생성·실행해야 하는 대량 복제 배치를 구축.

해결: Shell로 전체 파이프라인을 구성해 대상 테이블마다 ETL 작업을 자동 생성·실행하고, 완료 시 Oracle 원본 테이블의 STATUS='Y', PROC_DT(처리일자)를 갱신해 진행 상태를 추적.

트러블슈팅 - 데드락: 다수의 배치 프로세스가 동시에 같은 테이블의 상태 컬럼을 갱신하려 하면서 데드락(Deadlock)이 반복 발생.

해결 - Random Sleep: Shell 스크립트 상단에 RANDOM 기반 sleep 값을 두어 프로세스별 시작 시점을 분산, 동시 접근 확률을 낮춰 데드락을 해소.

수천 개 테이블 사전 검증을 위한
Bad Data 추출 쿼리 자동 생성 시스템

문제: 본이행 전, AS-IS 스테이징 DB의 수천 개 테이블에 대해 매핑 정의서 기준 검증 규격에 어긋나는 Bad Data를 사람이 일일이 쿼리로 찾아낼 수 없는 규모.

해결 - SQL을 만드는 SQL: 매핑 정의서(자료형, 자릿수, Not Null, 코드값 등 검증 규격)를 기준으로 **각 테이블/컬럼별 Bad Data 추출 쿼리 자체를 생성하는 쿼리**를 설계, 수천 개 검증 쿼리를 자동으로 산출.

실행: 생성된 쿼리를 Shell Script로 일괄 구동, 배치 프로그램화하여 수천 개 테이블을 정기적으로 검증하고, 에러 발생 시 로그를 까서 원인을 파악·조치.

Python으로 전환: 이후 검증 대상이 SQL*Plus의 LINESIZE 한계(32,767자)를 넘는 대용량 CLOB SQL로 확장되면서, 쿼리 생성·파일 출력 로직을 Shell에서 Python 스크립트로 직접 재작성해 대응. 이 과정에서 인코딩(UTF-8), 개행 문자, 경로 처리 등 실무 파일 자동화 노하우를 축적.

함수화: 날짜 형식 검증처럼 반복되는 로직은 `IS_DATE()` 같은 PL/SQL 사용자 정의 함수로 만들어 검증 쿼리 전반에서 재사용.

확장 가능성: 검증 규격을 기준으로 데이터 오류를 정기적으로 자동 탐지·기록하는 구조라, 데이터 정합성에 대한 내부통제(ITGC) 관점의 상시 모니터링 체계로도 적용할 수 있습니다.

Unpivot 압축과 ROWID 정렬 분할로 해결

문제 1 - 속도 저하: PROJECT 02의 검증 쿼리를 계속 운영하다 보니 컬럼 단위로 흩어진 검증 결과를 모으는 과정에서 속도 이슈 발생.

해결 1 - Unpivot 압축 통합: 컬럼 단위로 흩어져 있던 검증 로직을 **Unpivot 기법으로 압축·통합하는 프로그램을 Python으로 직접 개발**, 처리 단위를 대폭 줄임.

문제 2 - ORA-01555: 대용량 테이블 검증 시 ORA-01555 (snapshot too old)가 반복 발생.

해결 2 - ROWID 분할 + 물리 정렬: 작업을 **ROWID 범위로 나눠 분할 처리**하고, 이때 **물리적 블록 주소가 가까운 로우끼리 ORDER BY** 하여 랜덤 I/O를 줄이고 메모리 사용을 최소화.
